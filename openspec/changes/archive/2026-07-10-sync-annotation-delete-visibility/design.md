## Context

Today the sync layer only ever grows an annotation clip's `annotation_commands` list (append via `INSERT_CHILD`, or a same-size/edited replace via `REPLACE_ANNOTATION_COMMANDS` for text edits and drag-moves). Neither RV's native "Clear Frame" / "Clear All Frames on Timeline" / "Show Drawings" actions, nor xStudio's native "Delete all strokes" (Ctrl+D) / "Toggle annotation visibility" ('V') actions, are observed by the plugin at all:

- RV: `plugin.py`'s `init([...])` binding list has no entry for RV's internal `clear-paint` / `clear-all-paint` events, and `annotation_sync.on_graph_state_change` explicitly treats any `.order`-suffixed property change as something to skip.
- xStudio: `on_annotation_event` already receives every `AnnotationsUI` event (including `PaintClear`, `HideDrawings`, `ShowDrawings`) as a `(event_atom, annotation_atom, JsonStore)` tuple, but only uses it as a generic "go rescan bookmarks" trigger — the `data["event"]` string itself is never inspected, and the scan logic (`broadcast_local_bookmark`) assumes stroke/caption counts only grow, silently doing nothing when they shrink.

Both receivers' existing `REPLACE_ANNOTATION_COMMANDS` handling is already an authoritative overwrite (RV: reconcile-mode prune of any managed node whose uuid is absent from the incoming batch; xStudio: `bm.set_annotation(...)` full overwrite) — but both currently special-case an **empty** incoming command list as "caller said nothing, so touch nothing," which defeats using this path for a full clear.

## Goals / Non-Goals

**Goals:**
- A stroke/shape/text deleted locally (one of several on a frame) disappears on all peers.
- "Clear Frame" / "Clear All Frames on Timeline" (RV) and "Delete all strokes" (xStudio) empty the affected clip(s)' annotations on all peers.
- Toggling annotation visibility locally (RV's per-source "Show Drawings", xStudio's global 'V' hotkey) hides/shows annotations on all peers.
- Reuse the existing `REPLACE_ANNOTATION_COMMANDS` message and `display_settings` broadcast — no new wire message types, no `SyncEvent` schema changes.

**Non-Goals:**
- Per-annotation (per-bookmark) visibility. xStudio's `BookmarkDetail.visible_` exists in the C++ model but has no UI affordance today; RV has no per-frame visibility concept at all. This change only syncs the two *global* toggles that actually exist in each tool's UI.
- Syncing undo/redo of a clear. RV's `undo-paint`/`redo-paint` internal events are not bound by this change (see Open Questions).
- Removing the xStudio `Bookmark` object itself (`remove_bookmark`) when a clear empties it. The bookmark/note is left in place with empty `annotation_data`, matching xStudio's own local behavior ("if there is no text in the associated note it will also be removed" happens elsewhere and independently of what we sync).

## Decisions

### Resolve deleted items by uuid against our own OTIO state, not by re-deriving the sender's node/frame

RV's `clear-paint` event payload is just a pipe-joined list of stroke uuids (`sendInternalEvent("clear-paint", "uuid1|uuid2|...")`); `clear-all-paint`'s payload additionally interleaves the RV node name per uuid. Rather than re-deriving which OTIO clip/frame each RV node/frame maps to (a third implementation of logic that already exists, non-trivially, in `_find_paint_node_for_media` and friends), the delete handler looks up each deleted uuid directly against the local `sync_manager`'s Annotations tracks (a new small helper, e.g. `annotation_clip_guid_for_stroke_uuid`), removes it from that clip's stored `annotation_commands`, and broadcasts one `REPLACE_ANNOTATION_COMMANDS` per affected clip with the survivors. This keeps annotation-deletion decoupled from RV's frame-numbering math entirely, and naturally handles `clear-all-paint` (which can touch many clips across many sources at once) as N independent per-clip replaces.

*Alternative considered*: parse the RV node name to resolve media path → clip_guid → frame per deleted uuid (mirroring `_broadcast_annotation`'s existing lookup chain). Rejected as strictly more code for the same result, and another place that could drift from the existing lookup helpers if RV's node-naming conventions ever change.

### Detect xStudio deletion as a count *decrease* in the existing poll-scan, not a separate code path

`broadcast_local_bookmark` already reads `bm.annotation_data` every scan and compares against `count_annotation_commands(clip_guid, frame)` to find newly-added strokes/captions (`new_strokes = all_strokes[sent_strokes:]`). Deletion is the same read, with the opposite-sign case: if `len(all_strokes) < sent_strokes` (or captions), the local user removed something. In that case, instead of computing a delta, rebuild the *complete* surviving stroke/caption list (reusing `extract_caption_uuids`-style uuid continuity so surviving items keep their existing uuids) and broadcast it via `broadcast_replace_annotation_commands`. `PaintClear` already schedules this scan via the existing `on_annotation_event` → `_annotation_pending_time` path — the only new wiring needed is reading `data["event"]` to short-circuit noise events (`SetDisplayMode`, tool-switch, etc.) and letting the count-decrease branch do the rest.

*Alternative considered*: react to `PaintClear` directly and immediately zero out the tracked clip, without waiting for the debounced scan. Rejected: `PaintClear` fires before xStudio necessarily commits the emptied `annotation_data`, and the existing debounce/retry machinery (`_annotation_flush_retries`) already exists to handle exactly this kind of commit-lag — reusing it avoids a second, parallel timing workaround.

### Empty-payload hard-clear as an explicit branch, not a change to the shared reconcile inference

Both `rv_paint_applier.apply_specs` (reconcile mode) and xStudio's `refresh_annotation_bookmark` treat "nothing in the incoming batch" as "caller is silent on this kind, don't touch it" — required for the existing text-edit replace path, which always omits pen-stroke kinds on purpose. Rather than teaching `apply_specs` to distinguish "silent" from "authoritatively empty" (which would mean widening its signature and auditing every existing caller), the new delete/clear path adds one explicit branch on each receiver: when the rebuilt command list for a clip is completely empty, skip the kind-inferring reconcile/merge logic entirely and hard-clear directly (RV: `setStringProperty(order_prop, [], True)` on the resolved paint node/frame; xStudio: `bm.set_annotation(strokes=[], captions=[])` on the tracked bookmark). Existing partial-replace callers (text edits) are untouched and keep their current omission semantics.

### Visibility: one global `annotations_visible` flag on `display_settings`, applied session-wide in RV

xStudio's only user-facing visibility control (`annotations_visible_`, toggled by the 'V' hotkey, sending `HideDrawings`/`ShowDrawings`) is a single session-wide boolean — there is no per-bookmark toggle in the UI today. RV's only control (`<node>.paint.show`) is scoped to one `RVPaint` node (one media source). To keep one shared flag, RV's toggle is broadcast and applied the same way regardless of which node fired it: on send, broadcast `{"annotations_visible": bool}` merged into `display_settings` (mirroring how exposure/channel already merge into that same blob); on receive, apply it to *every* `RVPaint` node's `.paint.show`, not just the currently-active one. This is a real (and accepted, per proposal discussion) semantic widening: hiding one source's drawings in RV becomes "hide all drawings everywhere" for peers.

*Alternative considered*: per-clip `hidden` metadata, matching RV's actual scope and xStudio's underlying (if UI-unexposed) `BookmarkDetail.visible_` exactly. Rejected for this change: no widening, but requires new per-clip state (parallel to `annotation_commands`) and xStudio's single global hotkey would need to fan out to "mark every clip hidden" instead of flipping one flag — more moving parts for a UI gesture that, in both tools today, is not actually per-annotation.

### Applying a remote visibility toggle in xStudio via the already-obtained `AnnotationsUI` plugin handle

`ori_sync_plugin.py` already does `ann_plugin = self.get_plugin("AnnotationsUI")` to subscribe to its events, but discards the handle immediately after `subscribe_to_plugin_events`. This change retains it (`self._ann_ui_plugin`) and, on receiving a remote `annotations_visible` change, calls `self._ann_ui_plugin.set_attribute("Visibility", bool)` — the same `get_attribute`/`set_attribute` pattern `display_sync.py` already uses against the viewport module (`vp.set_attribute("Fit (F)", "Off")`), just against a different module instance. See Open Questions — this specific call has not been exercised against `AnnotationsUI` in this codebase yet and should be spiked early.

## Risks / Trade-offs

- **[Risk] RV's per-source hide broadcasting as global hide is a real UX widening** (see Decision above) → Mitigation: called out explicitly in the proposal and accepted; if it proves surprising in practice, the per-clip alternative is documented above as the fallback design.
- **[Risk] `AnnotationsUI.set_attribute("Visibility", ...)` may not behave like the viewport's `set_attribute`** (untested in this codebase against a non-viewport plugin module) → Mitigation: spike this call in isolation before wiring the full receive path; if it doesn't work, fall back to whatever mechanism `colour_picker_hide_drawings_` (a sibling attribute already toggled alongside `annotations_visible_` in the C++ handler) implies is the supported external-facing route.
- **[Risk] `clear-all-paint` can touch many clips in one user gesture** (every annotated frame across every source) → Mitigation: broadcast one `REPLACE_ANNOTATION_COMMANDS` per affected clip in a tight loop; no batching/new message type planned for v1 since this is a rare, deliberate user action, but worth watching for network chatter in testing.
- **[Risk] Race between a local edit and a remote clear landing on the same clip** → Mitigation: none beyond what already exists (last-write-wins via the existing `_last_sent_replace_sig` dedup and the manager's own OTIO-as-source-of-truth model); not worse than the existing text-edit replace path's behavior.

## Migration Plan

No data migration. Purely additive event bindings and a new optional `display_settings` field (`annotations_visible`, defaults to visible/`True` when absent so old snapshots and older peers are unaffected). Roll out by deploying both plugins together; a mixed old/new peer session degrades gracefully — an old peer simply never broadcasts or reacts to deletes/visibility, exactly today's behavior, while a new peer still correctly reflects its own local deletes/visibility to other new peers.

## Open Questions

- Does `AnnotationsUI.set_attribute("Visibility", bool)` actually work from Python the way `vp.set_attribute(...)` does for the viewport? Needs a quick spike (see Risks).
- Should RV's `undo-paint`/`redo-paint` (which can *restore* a cleared frame) be wired in this change, or left for a follow-up? Proposal and this design assume follow-up; flagging so it isn't lost.
- `clear-all-paint`'s payload also clears `sourceFrame(frame)` duplicates (RV's handling of frames that alias to the same source timecode) — need to confirm this doesn't produce duplicate/conflicting `REPLACE` broadcasts for the same clip when both a frame and its source-frame alias are processed.

## Findings From Manual Testing (post-implementation)

Six rounds of live testing against real RV and xStudio instances surfaced several issues the design didn't anticipate — none in the core architecture (uuid-based resolution, REPLACE reuse, empty-hard-clear branch), all in host-specific plumbing. All are now fixed and confirmed working: single-frame clear, multi-frame "Clear All Frames on Timeline" with several strokes per frame, xStudio's Ctrl+D clear, and visibility toggling in both directions (RV↔xStudio).

**RV pen strokes never had a resolvable uuid at all.** The very first test showed `on_clear_paint` firing correctly (RV's own `clear-paint` event binding works fine — an early scare that it wasn't binding turned out to be a logging gap, not a real problem) but resolving 0 of 1 deleted uuids. Root cause: RV's native annotate_mode auto-assigns *some* `.uuid` to every pen stroke for its own undo bookkeeping, but this sync plugin's send path never wrote *its own* broadcast uuid back onto that property (an explicit pre-existing design choice — pens tracked uuid in-memory only, since nothing used to need to read it back). The two identifier spaces were simply disjoint. **Fixed**: `_construct_annotation_events` (send path) and `_apply_annotation` (receive path, for strokes originating remotely) both now persist the broadcast uuid onto RV's `.uuid` property, mirroring what text/shape annotations already did.

**Multiple strokes on one frame exposed a second, deeper bug**: a *single* stroke gesture was generating a fresh random uuid on nearly every mouse-move tick (`on_graph_state_change`'s pen tracking minted a new uuid whenever `_next_stroke_uuid` had already been consumed — harmless before since nothing read it back, but now an intermediate tick's uuid could get permanently committed as a phantom duplicate "stroke" once RV's live property moved on to a newer value, forever unresolvable by any later clear). Worse, RV's own pen-up detection (likely the `pointer--leave` binding) was observed firing prematurely mid-drag on the *same* pen component, splitting one physical stroke into two separately-committed uuids. **Fixed** in two layers: (1) reuse the in-flight `_pending_stroke`'s uuid for the same (node, component) instead of minting a new one per tick; (2) when `_pending_stroke` doesn't match (including after a premature flush reset it), check RV's own `.uuid` property first and reuse it if already set, only falling back to a fresh uuid when the component has genuinely never been broadcast. Confirmed working across multiple strokes per frame, multiple frames, full "Clear All Frames on Timeline".

**xStudio's chosen event hook (`on_annotation_event`, subscribed via `plugin_events_group_atom()`) never fires at all — a pre-existing gap, not something this change broke.** `AnnotationsUI::send_event()` sends `PaintClear`/`HideDrawings`/`ShowDrawings` as a direct point-to-point message to the AnnotationsCore actor, not to AnnotationsUI's own event group (the pre-existing `"[TEST annotation_atom]"` log comment was already probing this same suspicion). Tracing AnnotationsCore's C++ side found the real, working signal: `clear_annotation()` broadcasts a **3-tuple** `(event_atom, annotation_data_atom, anno_ptr)` to `live_edit_event_group_` — the *same* group `on_core_annotation_event` already successfully subscribes to for live strokes, just one element shorter than the 4/5-tuple shape its guard required (silently rejecting it). **Fixed**: the guard now accepts `len(data) >= 3` and schedules the existing debounced flush on an exact 3-tuple, without needing to parse the payload — `broadcast_local_bookmark`'s count-decrease detection (already implemented) does the rest. Confirmed working (`strokes 4->0`, `strokes 2->0` observed clearing correctly in testing), though the fast 3-tuple path specifically wasn't confirmed to have fired in the successful test (the 30s periodic fallback scan may have been what actually caught it) — low priority to chase further since the outcome is correct either way.

**xStudio visibility apply was calling the wrong mechanism.** `AnnotationsUI::attribute_changed()` has no branch at all for the `Visibility` attribute directly — only for `action_attribute_`. Setting `Visibility`'s plain value directly (the original implementation) is a complete no-op: it changes a cosmetic stored value with zero side effects, since only `action_attribute_` being set to `["HideVisibility"]`/`["ShowVisibility"]` (what the real 'V' hotkey does) triggers the `send_event(...)` call that actually notifies `AnnotationsCore::hide_all_drawings_`. **Fixed**: `apply_display_state` now sets `action_attribute_`'s value to the appropriate one-element list instead. Confirmed working in both directions (RV→xStudio and xStudio→RV). The `_read_annotations_visible` read path (RV side) also had a bug: it scanned all `RVPaint` nodes for "any that has `.paint.show` set," which could pick a stale node once a remote broadcast had already written the property session-wide to every node — fixed to resolve the specific currently-viewed node via `metaEvaluateClosestByType`, matching the pattern `_find_paint_node_for_media` already uses.

All fixes are additive/corrective within the send/receive paths this change already owns; no architectural decisions from the original design changed.
