## 7. Verification

- [x] 7.1 Run a `draw_annotation` + `capture_frame` + comparison end-to-end for at least one shape test and confirm the comparison reports a sensible measured thickness (sanity-check the numbers, not just "it ran without error"). Ran all three live (`rect`/`ellipse`/`arrow`): rect expected 5.40px measured 6.69px (offset +1.29px); ellipse expected 5.40px measured 6.69px (offset +1.29px, centroid 3.21px); arrow expected 17.28px measured 20.68px (offset +3.40px, centroid 0.68px) — all sensible, all within the 4px default tolerance. Also caught and fixed a real bug during this: the initial `compute_color_weights` dot-product scoring (borrowed from `compare_testchart.py`'s hand-picked hardcoded weight vectors) was mathematically unsound for arbitrary mid-saturation colours — it scored the annotation's *own exact colour* negative for `[0.65, 0.25, 0.2]`, silently reporting "not found" despite the annotation rendering correctly (verified visually). Replaced with a distance-based match (`score = max(0, 1 - ||pixel - colour|| / tolerance)`), guaranteed maximal at an exact match.
  - **Follow-up 1**: the three live runs above only ever called `capture_frame` on the *xStudio* peer (`driver: openrv, peer: xstudio` in all three shape tests) — the OpenRV capture path (`sessionGLView()` grab, `setViewSize`, shiboken wrapping) was never actually exercised. Verified it directly with a standalone script driving OpenRV alone: `capture_frame` succeeded but returned a **3840×2160** image despite requesting 1920×1080 (HiDPI/Retina 2x scaling on `.grab()`) — exactly the scenario design D2 anticipated; `measure_shape_border`'s read-actual-dimensions approach handled it correctly, projecting geometry against the real 3840×2160 and landing exactly on the rendered box (bbox matched to the pixel). This same run (using an intentionally thick 0.02 border to stress-test) also caught a second real bug: the border renders *entirely inside* the `min`/`max` boundary line (not centered/straddling it), and the fixed `HALF_WIDTH=20` cross-section sampling window silently truncated anything thicker than ~20px, under-reporting a 43.2px-thick border as 20.2px. Fixed by scaling `half_width` with `expected_thickness_px` in `measure_shape_border` (`max(HALF_WIDTH, ceil(expected_thickness_px) + HALF_WIDTH)`) — re-measured at 42.68px (offset −0.52px). Re-ran all three production-scale tests after the fix to confirm no regression.
  - **Follow-up 2** (user feedback: "expecting to see two png files...   - **Follow-up 3** (user request: add visual checks for the pen tests too): extended `visual_check: true` to `openrv_draws_pen_xstudio_verifies` and `xstudio_draws_pen_openrv_verifies`. Pen needed a new `_edges_for_kind("pen", ...)` case (a plain straight segment, no arrowhead) and, since the two pen tests have different drivers, a driver-aware geometry lookup: `annotation_assertions.shape_geometry_for_driver(kind, driver_name)` — RV's raw paint-property points are already OTIO-normalized-equivalent for position (per `coords`'s docstring) so `DEFAULT_SHAPE_GEOMETRY` applies directly when RV is the driver, but xStudio's native stroke coordinates need the same `x_otio = x_xs * aspect_half` conversion `xs_annotation_codec` applies, so a new `_xstudio_pen_geometry()` covers the xStudio-driven case. This surfaced two more real, pre-existing bugs, both caught only because visual_check checks *rendered pixels* rather than just numeric round-trip:
    - **Colour bug**: `xstudio_hook.py::_draw_xstudio_annotation`'s synthetic stroke dict used `"type": "Brush"` (not a value the production codec recognises — it only ever writes `"color"`/`"erase"`) and omitted the legacy `"r"`/`"g"`/`"b"` keys the production `xs_annotation_codec.sync_events_to_xs_strokes` always populates alongside `"colour"` for backward compatibility. Fixed by scaling the tolerance with the expected thickness (`max(tolerance_px, 0.3 * expected_thickness_px)`) in `runner.py::_capture_and_measure`. Re-ran both pen tests (and all three shape tests, to confirm no regression) multiple times each across the observed resolution variability — all consistently pass.
